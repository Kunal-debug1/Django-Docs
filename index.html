Django is a high-level Python web framework that lets you build powerful, secure, and scalable web applications quickly — without reinventing the wheel.

What is Django?

Django is a free, open-source web framework written in Python. It was created in 2005 by Adrian Holovaty and Simon Willison at a newspaper company in Kansas — that's why it focuses on rapid development.

Its motto: "The web framework for perfectionists with deadlines."

Set up a professional Django development environment from scratch. Follow each step carefully.

bash

python --version
# Output: Python 3.11.x or higher
2
Create a virtual environment
A virtual environment keeps your project dependencies isolated from other projects.
bash
# Create a virtual environment named "venv"
python -m venv venv

# Activate it — Windows
venv\Scripts\activate

# Activate it — Mac / Linux
source venv/bin/activate

# Your terminal will show: (venv) before the prompt
3
Install Django
bash
pip install django

# Verify
python -m django --version
# Output: 5.x.x

Create your first project
bash
django-admin startproject myproject
cd myproject
python manage.py runserver
✅ Visit http://127.0.0.1:8000 in your browser. You should see the Django rocket launch page!

Create an app

Then register it in settings.py:

    'django.contrib.auth',
    ...
    'blog',  # ← Add your app here

Django Project Structure

Understanding every file in a Django project is your first real step. Let's decode it all.

file tree
myproject/
├── manage.py               # CLI entry point for all Django commands
├── myproject/
│   ├── __init__.py         # Marks directory as a Python package
│   ├── settings.py         # ALL configuration lives here
│   ├── urls.py             # Root URL patterns (main router)
│   ├── wsgi.py             # Entry point for WSGI web servers (prod)
│   └── asgi.py             # Entry point for async/ASGI servers
└── blog/                   # Your app
    ├── migrations/         # Auto-generated DB schema changes
    │   └── 0001_initial.py
    ├── templates/          # HTML template files (recommended)
    ├── static/             # CSS, JS, images for this app
    ├── __init__.py
    ├── admin.py            # Register models with admin panel
    ├── apps.py             # App configuration class
    ├── models.py           # Database models (tables)
    ├── views.py            # Business logic — handles requests
    ├── urls.py             # App-level URL patterns
    ├── forms.py            # Form classes (you create this)
    ├── serializers.py      # DRF serializers (you create this)
    └── tests.py            # Unit and integration tests

Chapter 04

URLs & Views

Every web request starts with a URL and ends with a View returning a response. Learn how Django routes traffic.

URL routing

Django uses urls.py to match incoming URLs to Python functions (views). You include app URLs in the main project URLs file.

python — myproject/urls.py (root)
from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    path('admin/', admin.site.urls),
    path('',        include('blog.urls')),   # homepage
    path('api/',    include('api.urls')),    # API routes
    path('users/',  include('users.urls')),  # auth routes
]
python — blog/urls.py (app)
from django.urls import path
from . import views

urlpatterns = [
    path('',                    views.home,        name='home'),
    path('about/',               views.about,       name='about'),
    path('post/<int:pk>/',       views.post_detail, name='post_detail'),
    path('post/<slug:slug>/',    views.post_slug,   name='post_slug'),
    path('post/create/',         views.create_post, name='create_post'),
]

Function-Based Views (FBV)

python — blog/views.py
from django.shortcuts import render, redirect, get_object_or_404
from django.contrib.auth.decorators import login_required
from .models import Post

def home(request):
    # Get all published posts
    posts = Post.objects.filter(published=True).order_by('-created_at')
    return render(request, 'blog/home.html', {'posts': posts})

def post_detail(request, pk):
    # Returns 404 if post doesn't exist
    post = get_object_or_404(Post, pk=pk, published=True)
    return render(request, 'blog/detail.html', {'post': post})

@login_required  # Only logged-in users can access this
def create_post(request):
    if request.method == 'POST':
        # Process form data
        return redirect('home')
    return render(request, 'blog/create.html')

Returning different response types

python
from django.http import HttpResponse, JsonResponse
from django.shortcuts import redirect

# Plain HTML
return HttpResponse("<h1>Hello World</h1>")

# Template with context
return render(request, 'template.html', {'key': value})

# JSON response (for APIs)
return JsonResponse({'status': 'ok', 'data': [1, 2, 3]})

# Redirect to another URL
return redirect('home')              # by name
return redirect('/some/path/')       # by path
return redirect('post_detail', pk=5) # with args
Chapter 05

Models & ORM

Models are Python classes that map directly to database tables. Django's ORM lets you work with data using pure Python — no SQL required.

Defining models

python — blog/models.py
from django.db import models
from django.contrib.auth.models import User
from django.urls import reverse

class Category(models.Model):
    name = models.CharField(max_length=100)
    slug = models.SlugField(unique=True)

    def __str__(self):
        return self.name

class Tag(models.Model):
    name = models.CharField(max_length=50)

class Post(models.Model):
    title      = models.CharField(max_length=200)
    slug       = models.SlugField(unique=True)
    body       = models.TextField()
    author     = models.ForeignKey(User, on_delete=models.CASCADE,
                                   related_name='posts')
    category   = models.ForeignKey(Category, on_delete=models.SET_NULL,
                                   null=True, blank=True)
    tags       = models.ManyToManyField(Tag, blank=True)
    image      = models.ImageField(upload_to='posts/', blank=True)
    published  = models.BooleanField(default=False)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
    views      = models.PositiveIntegerField(default=0)

    class Meta:
        ordering = ['-created_at']
        verbose_name_plural = 'posts'

    def __str__(self):
        return self.title

    def get_absolute_url(self):
        return reverse('post_detail', args=[self.slug])

Model field types

ORM — CRUD operations

python — Django shell
# ---- CREATE ----
post = Post.objects.create(
    title='My First Post', body='Hello Django!', author=user
)

# ---- READ ----
Post.objects.all()                           # All posts
Post.objects.filter(published=True)         # Published only
Post.objects.filter(author__username='ali') # Filter by related field
Post.objects.get(pk=1)                     # Single — raises error if not found
Post.objects.first()                         # First result
Post.objects.count()                         # Count rows

# ---- UPDATE ----
post = Post.objects.get(pk=1)
post.title = 'Updated Title'
post.save()

Post.objects.filter(published=False).update(published=True)  # bulk update

# ---- DELETE ----
post.delete()
Post.objects.filter(published=False).delete()  # bulk delete

# ---- ADVANCED ----
Post.objects.filter(title__icontains='django')    # case-insensitive search
Post.objects.filter(created_at__year=2024)         # date filtering
Post.objects.select_related('author').all()        # avoids N+1 (FK)
Post.objects.prefetch_related('tags').all()        # avoids N+1 (M2M)
Post.objects.order_by('title')                     # ascending
Post.objects.order_by('-created_at')               # descending
Post.objects.values('title', 'author__username')   # dict output
⚠️ Common mistake: Using .get() without a try/except. It raises DoesNotExist if nothing is found. Use get_object_or_404() in views instead — it returns a clean 404 page automatically.

Chapter 07

Django Forms

Forms handle user input safely — Django provides automatic CSRF protection, validation, and error messages.

ModelForm (the most common approach)

python — blog/forms.py
from django import forms
from .models import Post

class PostForm(forms.ModelForm):
    class Meta:
        model = Post
        fields = ['title', 'body', 'category', 'image']
        widgets = {
            'title': forms.TextInput(attrs={
                'class': 'form-control',
                'placeholder': 'Enter post title...',
            }),
            'body': forms.Textarea(attrs={
                'rows': 8,
                'class': 'form-control',
            }),
        }

    # Custom field-level validation
    def clean_title(self):
        title = self.cleaned_data['title']
        if len(title) < 5:
            raise forms.ValidationError("Title must be at least 5 characters.")
        return title

    # Cross-field validation
    def clean(self):
        cleaned_data = super().clean()
        return cleaned_data

Handling forms in a view

python — blog/views.py
from django.contrib import messages
from .forms import PostForm

@login_required
def create_post(request):
    if request.method == 'POST':
        form = PostForm(request.POST, request.FILES)  # FILES for uploads
        if form.is_valid():
            post = form.save(commit=False)  # don't save yet
            post.author = request.user        # set author manually
            post.save()
            form.save_m2m()                   # save ManyToMany (tags)
            messages.success(request, 'Post created!')
            return redirect('post_detail', pk=post.pk)
    else:
        form = PostForm()

    return render(request, 'blog/create.html', {'form': form})

Form in template

html
<form method="POST" enctype="multipart/form-data">
  {% csrf_token %}

  {% for field in form %}
    <div class="field-group">
      {{ field.label_tag }}
      {{ field }}
      {% if field.errors %}
        <ul class="errors">
          {% for error in field.errors %}
            <li>{{ error }}</li>
          {% endfor %}
        </ul>
      {% endif %}

  {% endfor %}

  <button type="submit">Save Post</button>
</form>
🚫 Always include {% csrf_token %} in every POST form. Django will return a 403 Forbidden error without it — this is a security feature, not a bug.

Registration view

python — users/views.py
from django.contrib.auth import login, logout
from django.contrib.auth.forms import UserCreationForm

def register(request):
    if request.user.is_authenticated:
        return redirect('home')

    if request.method == 'POST':
        form = UserCreationForm(request.POST)
        if form.is_valid():
            user = form.save()
            login(request, user)
            messages.success(request, f'Welcome, {user.username}!')
            return redirect('home')
    else:
        form = UserCreationForm()
    return render(request, 'users/register.html', {'form': form})

from django.contrib import admin
from .models import Post, Category, Tag

@admin.register(Post)
class PostAdmin(admin.ModelAdmin):
    list_display   = ['title', 'author', 'category', 'published', 'created_at']
    list_filter    = ['published', 'category', 'created_at']
    search_fields  = ['title', 'body']
    prepopulated_fields = {'slug': ('title',)}
    list_editable  = ['published']
    date_hierarchy = 'created_at'
    ordering       = ['-created_at']
    readonly_fields = ['created_at', 'updated_at']

    fieldsets = (
        ('Content',   {'fields': ('title', 'slug', 'body', 'image')}),
        ('Meta',      {'fields': ('author', 'category', 'tags')}),
        ('Publish',   {'fields': ('published',)}),
        ('Timestamps',{'fields': ('created_at', 'updated_at'), 'classes': ('collapse',)}),

@admin.register(Post)
class PostAdmin(admin.ModelAdmin):
    actions = ['make_published']

    @admin.action(description='Mark selected posts as published')
    def make_published(self, request, queryset):
        count = queryset.update(published=True)
        self.message_user(request, f'{count} posts marked as published.')

Serializer

python — blog/serializers.py
from rest_framework import serializers
from .models import Post

class PostSerializer(serializers.ModelSerializer):
    author_name = serializers.CharField(
        source='author.username', read_only=True
    )
    url = serializers.HyperlinkedIdentityField(
        view_name='post-detail'
    )

    class Meta:
        model = Post
        fields = ['id', 'url', 'title', 'body',
                  'author_name', 'created_at', 'published']
        read_only_fields = ['id', 'created_at']

ViewSet (full CRUD in ~12 lines)

python — blog/views.py
from rest_framework.viewsets import ModelViewSet
from rest_framework.permissions import IsAuthenticatedOrReadOnly
from rest_framework.decorators import action
from rest_framework.response import Response

class PostViewSet(ModelViewSet):
    queryset = Post.objects.filter(published=True)
    serializer_class = PostSerializer
    permission_classes = [IsAuthenticatedOrReadOnly]

    def perform_create(self, serializer):
        serializer.save(author=self.request.user)

    @action(detail=True, methods=['post'])
    def publish(self, request, pk=None):
        post = self.get_object()
        post.published = True
        post.save()
        return Response({'status': 'published'})

Performance Optimization

Make your Django app fast. The most common bottleneck is the database — these techniques will solve 90% of performance issues.

The N+1 query problem

The most common Django performance issue. When you loop through objects and access a related field, Django makes one query per object — that's N+1 queries.

python — bad vs good
# ❌ BAD — 1 + N queries (N = number of posts)
posts = Post.objects.all()
for post in posts:
    print(post.author.username)  # DB query each time!

# ✅ GOOD — 2 queries total (JOIN)
posts = Post.objects.select_related('author').all()
for post in posts:
    print(post.author.username)  # No extra query

# ✅ GOOD — M2M (prefetch in separate query)
posts = Post.objects.prefetch_related('tags').all()

Query optimization techniques

python
# Load only needed fields
Post.objects.values('id', 'title')
Post.objects.only('title', 'created_at')
Post.objects.defer('body')  # Skip large body field

# Add database indexes
class Post(models.Model):
    title = models.CharField(max_length=200, db_index=True)
    created_at = models.DateTimeField(auto_now_add=True, db_index=True)

# Aggregate without loading objects
from django.db.models import Count, Avg, Sum

Post.objects.aggregate(total=Count('id'))
Post.objects.values('category').annotate(count=Count('id'))

# Efficient existence check
if Post.objects.filter(author=user).exists():  # ✅ fast
    ...
if Post.objects.filter(author=user).count() > 0:  # ❌ slower
    ...

Interview Preparation

Common Django interview questions with detailed answers. Click each question to reveal the answer.

Core Django

What is Django's MVT pattern? How is it different from MVC? ▶
Django follows MVT: Model (data layer — ORM), View (business logic — handles requests, similar to MVC's Controller), and Template (presentation — renders HTML, similar to MVC's View). The difference is that in Django the framework itself acts as the Controller by routing URLs to Views.
What is middleware in Django? Give examples. ▶
Middleware is code that processes every request/response globally, before it reaches a view or after it leaves. Examples: SecurityMiddleware (HTTPS redirect), SessionMiddleware (sessions), AuthenticationMiddleware (attaches user to request), CsrfViewMiddleware (CSRF protection). Custom middleware can add headers, log requests, rate limit, etc.
What is the difference between select_related and prefetch_related? ▶
select_related uses a SQL JOIN — best for ForeignKey and OneToOne fields. It fetches everything in a single query. prefetch_related performs a separate query and joins in Python — best for ManyToMany and reverse ForeignKey relations. Use select_related('author') for a post's author, and prefetch_related('tags') for a post's tags.
How does Django's ORM prevent SQL injection? ▶
Django's ORM parameterizes all queries automatically. When you write Post.objects.filter(title=user_input), Django passes user_input as a separate parameter to the database driver — it's never concatenated into the SQL string. The database handles escaping. Never use raw SQL with user input unless you use Django's params argument.
When would you use signals? What are the risks? ▶
Signals are good for: creating related objects on save (Profile on User create), sending notifications, updating cache. Risks: they make code harder to trace (action-at-a-distance), can cause infinite loops if you're not careful, and can slow things down if overused. Don't use signals for business-critical logic — use them for side effects that are truly decoupled.
What's the difference between abstract and proxy models? ▶
Abstract models have no database table — they exist only to share fields and methods between child models. Proxy models share the exact same database table as the parent but let you change Python behavior (custom methods, ordering, default manager). Use abstract for "template" models; use proxy to create different representations of the same data.

DRF & API Questions

What is the difference between Serializer and ModelSerializer? ▶
ModelSerializer auto-generates fields from a model — you just specify model and fields. It also auto-generates validators and handles create/update. Serializer requires manual field definition — more flexible but more code. Use ModelSerializer for standard CRUD APIs; use base Serializer for complex, non-model data structures.
How does JWT authentication work in Django? ▶
User logs in with credentials → server returns two tokens: access token (short-lived, 5-60 min) and refresh token (long-lived, days/weeks). Client stores both, sends access token in Authorization: Bearer <token> header on every request. When access token expires, client sends refresh token to get a new access token — no re-login needed. Server validates tokens by signature, no database lookup needed.
How would you optimize a slow Django page? Walk through your process. ▶
1) Install Django Debug Toolbar — identify the slow queries. 2) Fix N+1 queries with select_related/prefetch_related. 3) Add db_index=True on frequently filtered fields. 4) Use .only() or .values() to avoid loading unused fields. 5) Cache expensive queries with Redis (cache.set). 6) Cache full views with @cache_page. 7) Use .exists() instead of .count() > 0. 8) Paginate large result sets.
✅ Interview tip: Always explain the "why" behind your answer, not just the "what". For example, don't just say "use select_related" — explain that it avoids N+1 queries by using a SQL JOIN.
